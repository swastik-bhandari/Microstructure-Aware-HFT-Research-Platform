Layer 1 — C++ Limit Order Book Engine
Weeks 1–5 · ~140h
1.1
Core order book data structure
Price-level map with time-priority queues for bid and ask sides
~24h

Implementation steps

  • Define Order struct: id, side, price, qty, timestamp (nanosecond)
  • Use std::map<Price, std::deque<Order>> for bids (descending) and asks (ascending)
  • Implement add_order(), cancel_order(), modify_order()
  • Write unit tests: correct best bid/ask after each mutation
  • Benchmark: time 1M insertions, target <200ns amortised per op
C++17std::mapstd::dequeGoogle Test
Risk: low — well-understood data structure
1.2
Price-time priority matching engine
Continuous matching of incoming orders against resting orders
~28h

Implementation steps

  • Implement match(): walk resting queue at best price, fill in FIFO order
  • Handle partial fills: reduce resting order qty, emit fill events
  • Market orders: aggressively consume liquidity until filled or book exhausted
  • Limit orders: match if marketable, else rest at price level
  • Emit structured FillEvent{aggressor_id, passive_id, price, qty, ts}
  • Unit test: cross orders, partial fills, self-trade prevention
C++17event emissionprice-time priority
Risk: medium — edge cases in partial fill sequencing
1.3
Lock-free ring buffer + memory pool
Eliminate heap allocation and scheduling jitter under high event rates
~30h

Implementation steps

  • Implement SPSC lock-free ring buffer using std::atomic head/tail
  • Pre-allocate a slab memory pool for Order objects — no runtime new/delete
  • Cache-line align all hot structs (alignas(64))
  • CPU-pin matching engine thread via pthread_setaffinity_np
  • Benchmark with Google Benchmark: verify >1M events/sec sustained throughput
  • Profile with perf stat: check cache-miss rate, branch mispredictions

Risk: high — hardest C++ feature; allow 1 extra week buffer

Layer 2 — Microstructure Feature Engine
Weeks 6–7 · ~70h
2.1
Bid-ask spread + order book imbalance (OBI)
Core price-level signals computed on every LOB update
~14h
  • Spread: (best_ask - best_bid) / mid_price, normalised; update on every quote change
  • OBI: bid_vol / (bid_vol + ask_vol) at best quote level
  • Extend OBI to N levels: weighted sum over top-5 levels
  • Test: on a flat synthetic book OBI should be exactly 0.5
  • Benchmark: all 6 features within 1µs of LOB update (use rdtsc)

Queue position estimator
Estimated rank of a resting limit order within its price queue
~12h
  • Track cumulative volume ahead of our order at its price level
  • Update on each fill/cancel at that level
  • Derive fill probability: P(fill) ≈ exp(-λ · queue_pos / arrival_rate)
  • Validate on a controlled synthetic stream with known arrival rate

Layer 3 — Research Models & RL Agent
Weeks 8–13 · ~200h
3.1
TWAP, VWAP, and POV baselines
Classical execution strategies as benchmarks
~20h
  • TWAP: divide target qty into T equal slices, submit one per interval
  • VWAP: compute expected volume profile from historical data; weight slices proportionally
  • POV: at each step submit participation_rate × observed_market_volume
  • Wrap all three in a common Strategy interface: act(state) → Action
  • Run each against 1000 synthetic episodes; record all metrics to CSV
PythonPandascommon Strategy interface
3.2
ARIMA spread prediction + Kalman filter
Statistical models as intermediate complexity baselines
~22h
  • Fit ARIMA(1,0,1) on spread time series; predict next tick spread
  • Kalman filter: track hidden mid-price trend state; use innovation as signal
  • Route strategy: if spread predicted to narrow → post limit; else → market order
  • Compare to TWAP baseline on same synthetic episodes
statsmodelspykalman
3.3
CNN-LSTM mid-price direction model
Deep learning feature model producing directional signal for RL state
~40h
  • Input: rolling window of LOB snapshots, shape [T, 40] (top-10 bid/ask price+vol)
  • Architecture: 2× Conv1D (32 filters, k=3) → LSTM(64) → Dense(3) (up/flat/down)
  • Label: mid-price direction over next 5 ticks
  • Train/val split by episode; report accuracy, precision, recall per class
  • Export model.pt; wrap in DirectionalSignal class that returns P(up) scalar
  • Add P(up) as feature 7 in the RL state vector

3.4
Gymnasium RL environment wrapper
Wraps Layers 1–2 into a standard Gym interface for PPO training
~28h
  • Subclass gymnasium.Env; define observation_space and action_space
  • Observation: Box(shape=(N_features,)) — all normalised features

  • Test with stable_baselines3.common.env_checker.check_env()

3.5
PPO execution agent (training)
Proximal Policy Optimisation agent trained to minimise implementation shortfall
~50h
  • Start with SB3's PPO as scaffold; customise actor-critic network heads
  • Actor: MLP(128,128) → logits over 3 actions
  • Critic: MLP(128,128) → scalar V(s)
  • Hyperparameter sweep: lr ∈ {1e-4, 3e-4}, clip ε ∈ {0.1, 0.2}, entropy coeff
  • Train for 10M steps; log to TensorBoard (episode reward, entropy, value loss)
  • Checkpoint every 500k steps; keep best by eval episode shortfall
  • Evaluate: 1000 held-out episodes per checkpoint; plot learning curves

Scope cuts if behind

Definition of done
  • C++ engine: >1M events/sec on single core, unit tests green
  • Features: all 6 within 1µs, validated on controlled data
  • PPO: statistically significant improvement over TWAP on slippage
  • Platform: reproducible end-to-end in Docker with <5 commands
